 # 一、引言
 
 ## 1. 研究背景与意义
 短视频平台在移动互联网时代已成为主流内容形态之一。与图文系统相比，短视频系统在工程上同时面临**大文件上传**、**音视频处理（封面/元数据/多码率转码）**、**低延迟播放与分发**、**高频互动**以及**内容治理与安全控制**等多方面挑战。
 
 本项目（BS01 / VidSprout）是一个面向“用户上传—异步处理—多端播放—推荐分发—互动运营”的一体化短视频系统，后端采用 Django + DRF 提供 REST API，使用 Celery 进行异步任务调度，通过 FFmpeg/ffprobe 实现音视频处理，使用 PostgreSQL 作为核心业务数据库并配合索引/约束优化。
 
 研究意义主要体现在：
 - **工程实践价值**：在单机本地磁盘（无 Nginx）部署约束下，仍能实现可用的视频上传、媒体访问与转码任务执行；同时具备向对象存储/CDN/转码集群演进的清晰路径。
 - **系统研究价值**：系统涵盖上传链路、异步处理、数据建模、缓存、推荐流接口、鉴权与安全控制等典型问题，可用于系统架构设计与性能/安全评估研究。
 - **教学与论文价值**：代码结构清晰，关键业务模块可通过“需求—设计—实现—测试”完整描述并形成可复现实验。
 
 ## 2. 国内外研究现状
 国内外短视频系统相关研究主要集中在：
 - **音视频处理与分发**：FFmpeg 转码参数优化、多码率自适应、HLS/DASH 流媒体协议、低延迟传输与边缘分发。
 - **推荐系统**：从传统协同过滤（UserCF/ItemCF、矩阵分解）发展到深度学习推荐（双塔、序列模型、Transformer），以及多阶段“召回-粗排-精排-重排”架构。
 - **数据库与检索优化**：关系数据库在一致性与复杂查询上的优势仍广泛应用；PostgreSQL 的 GIN 索引、pg_trgm 模糊匹配等在内容检索中常见。
 - **安全与治理**：JWT/OAuth2、限流与反滥用、内容审核、审计日志等逐步成为系统“可上线”能力的一部分。
 
 本项目选择相对成熟稳定的技术组合，以保证“首日可用性”和工程可维护性，并为后续算法与架构升级留出空间。
 
 ## 3. 研究目的与问题阐述
 本研究目标是设计并实现一个可运行的短视频系统原型，并围绕以下问题展开：
 1. **上传与处理稳定性**：如何在直传与分片上传两种模式下保证文件落盘一致性，并在上传完成后尽快获得封面、时长、分辨率等元数据。
 2. **异步转码与播放链路**：如何使用 Celery 调度 FFmpeg 实现 HLS 转码、多档位输出以及缩略图/VTT 生成，并将处理结果与数据库状态一致化。
 3. **推荐流服务能力**：如何实现可用的推荐流 API，在保证接口性能（分页、缓存）的同时融入互动信号，并具备扩展到协同过滤的技术路线。
 4. **安全与访问控制**：如何实现 JWT 鉴权与可撤销机制，以及上传会话归属校验、文件类型校验等关键安全策略。
 
 ## 4. 研究内容与技术路线
 技术路线遵循“需求分析—系统设计—核心实现—测试与优化—评估与展望”：
 - 在需求层面对用户端/管理端功能、性能与安全进行拆解。
 - 在设计层面采用 B/S 三层架构，后端 REST API + 异步任务 + PostgreSQL。
 - 在实现层面提供：JWT 鉴权、视频上传（直传/分片）、同步元数据与封面生成、Celery 任务转码与缩略图/VTT、推荐流接口与缓存。
 - 在测试与优化层面给出功能测试矩阵、性能测试方法与合理的模拟结果。

 # 二、相关技术概述
 
 ## 1. Python 语言简介
 Python 具备高开发效率、生态成熟与跨平台优势，适合快速构建 Web 服务与后台任务系统。本项目中 Python 主要用于：
 - Web API（Django + DRF）
 - 异步任务（Celery）
 - 通过 `subprocess` 调用 FFmpeg/ffprobe
 - 与 PostgreSQL/Redis 等基础设施集成
 
 ## 2. FFmpeg音视频处理工具介绍
 FFmpeg 是主流的开源音视频处理工具集，支持编解码、转封装、滤镜与流媒体协议。本项目的关键使用点包括：
 - 使用 **ffprobe** 探测元数据（宽高、时长）。
 - 使用 **ffmpeg** 截帧生成封面与缩略图序列。
 - 使用 **ffmpeg** 转码生成 HLS（多码率/多分辨率），并生成低清 MP4 作为 processing 阶段占位资源。
 
 ## 3. 后端框架介绍
 
 ### 3.1 Django 框架核心架构（MVT 模式）
 Django 采用 MVT 模式：
 - **Model**：定义数据结构、索引与约束（如 `Video`、`Like`、`Favorite`）。
 - **View**：承载 API 请求处理（本项目主要使用 DRF 的 `APIView`）。
 - **Template**：本项目前后端分离，模板使用较少。
 
 ### 3.2 框架核心功能（ORM、Admin 后台、安全机制）
 - **ORM 与迁移**：支持复杂查询与索引/约束建模。
 - **安全机制**：中间件、CSRF/CORS 配置、认证与权限体系。
 - **REST 支持**：DRF 提供统一的认证、权限、分页、异常处理与限流框架。
 
 本项目依赖文件中声明 `Django==6.0.1`（见 `requirements.txt`），DRF 版本范围为 `>=3.14,<4`。
 
 ## 4. 数据库技术概述
 
 ### 4.1 Postgres 数据库特性
 PostgreSQL 具备强一致性事务、丰富索引类型与扩展生态，适合短视频业务的复杂查询与统计场景。本项目中还启用了 `django.contrib.postgres` 支持 Trigram 相似度等能力。
 
 ### 4.2 数据存储优化技术（索引、范式设计）
 本项目通过：
 - 部分索引加速已发布视频列表
 - GIN + trigram 索引支持标题/描述模糊检索
 - 中间表 `VideoTag` 实现多对多关系并建立组合索引

 ## 2. 核心功能模块实现
 
 ### 2.1 用户认证与权限管理实现
 本项目使用 SimpleJWT，并实现了可撤销（强制下线）的自定义鉴权类 `JWTAuthenticationWithRevoke`。核心逻辑如下（来自 `backend/backend/auth.py`）：
 
 **清单 2：JWT 可撤销鉴权（`backend/backend/auth.py::JWTAuthenticationWithRevoke.get_validated_token`）**
 ```python
 class JWTAuthenticationWithRevoke(JWTAuthentication):                               # 自定义 JWT 鉴权：增加“按用户撤销 token”能力
     def get_validated_token(self, raw_token):                                       # 每个请求都会调用：在这里统一判断 token 是否可用
         token = super().get_validated_token(raw_token)                               # 标准校验：签名、exp、格式等，失败则抛 AuthenticationFailed
         try:                                                                         # 防御式解析 token payload
             user_id = str(token.get('user_id') or token.get('user') or '')           # 用户 ID：兼容不同 payload 字段命名
             iat = int(token.get('iat') or 0)                                         # iat(issued at)：签发时间戳（秒）
         except Exception:                                                            # 任一字段缺失/类型不对都走兜底
             user_id = ''                                                             # 无 user_id 则无法做 revoke 比较
             iat = 0                                                                  # iat=0 表示未知签发时间
         if user_id:                                                                  # 仅在能识别用户时启用 revoke 校验
             key = f"logout_after:{user_id}"                                         # 缓存 key：记录该用户“强制重新登录”的时间阈值
             val = cache.get(key)                                                     # 从 Redis/缓存读取阈值；不存在表示无需强制下线
             if val:                                                                  # val 存在：表示用户被强制下线过
                 try:
                     cutoff = int(val)                                                # cutoff：撤销阈值（秒）；所有 iat < cutoff 的 token 都作废
                 except Exception:
                     cutoff = 0                                                       # 异常兜底：避免错误配置导致误封
                 if iat and iat < cutoff:                                             # 若 token 签发时间早于撤销阈值，则 token 无效
                     raise AuthenticationFailed('凭证已失效，请重新登录')               # 返回 401：提示客户端重新获取 token
         return token                                                                 # 校验通过：交由 DRF 继续解析 user 并进入业务逻辑
 ```
 
 该机制使管理员或系统可通过写入缓存 `logout_after:{user_id}` 来使旧 token 立即失效，满足安全运营需求。
 
 ### 2.2 视频上传、处理与播放实现
 
 #### (1) 直传上传（POST /api/videos/upload/）
 直传上传在 `apps/videos/views.py::VideoUploadView` 中实现。其关键步骤包括：
 - 文件大小限制（`VIDEO_MAX_SIZE_BYTES`）
 - 后缀与 MIME 校验
 - 临时文件落盘 + 原子替换（避免写入中断导致半文件）
 - 同步探测元数据（ffprobe）并生成封面（ffmpeg）
 - 创建 Video 记录并异步提交 Celery 任务（缩略图/VTT、HLS 转码）
 
 关键代码片段如下（来自 `VideoUploadView.post`）：
 
 **清单 3：直传上传核心逻辑（`backend/apps/videos/views.py::VideoUploadView.post`）**
 ```python
 file = request.FILES.get('file') or request.FILES.get('video')                      # 读取上传文件：兼容两种字段名
 if not file:                                                                        # 未提供文件则拒绝
     raise ValidationError({'file': '未收到文件'})                                    # 返回 400：提示选择文件
 max_bytes = int(getattr(settings, 'VIDEO_MAX_SIZE_BYTES', 524_288_000))             # 最大允许大小（默认 500MB，可配置）
 if file.size and file.size > max_bytes:                                             # 大小超限
     raise ValidationError({'file': '视频文件过大'})                                  # 返回 400：避免占用过多磁盘与转码资源
 
 ext = os.path.splitext(file.name or '')[1].lower()                                 # 取扩展名并小写化
 allowed_exts = {'.mp4', '.mov', '.m4v', '.webm', '.mkv'}                            # 白名单：只允许常见容器格式
 if not ext or ext not in allowed_exts:                                              # 扩展名非法
     raise ValidationError({'file': '不支持的文件格式'})                              # 返回 400：避免后续 ffmpeg 处理失败
 
 vid = uuid.uuid4().hex                                                             # 生成随机 key：避免文件名冲突与路径注入
 video_rel = f"videos/{vid}{ext}"                                                   # 视频相对路径：存储于 MEDIA_ROOT/videos
 video_abs = os.path.join(settings.MEDIA_ROOT, video_rel)                            # 视频绝对路径：用于写入
 thumb_rel = f"videos/thumbs/{vid}.jpg"                                            # 封面相对路径：存储于 MEDIA_ROOT/videos/thumbs
 thumb_abs = os.path.join(settings.MEDIA_ROOT, thumb_rel)                            # 封面绝对路径：ffmpeg 输出位置
 
 tmp_abs = f"{video_abs}.uploading"                                                # 临时文件名：写入完成再原子替换，防半文件
 with open(tmp_abs, 'wb+') as dst:                                                   # 打开临时文件写入
     for chunk in file.chunks():                                                     # 流式写入：避免一次性占用内存
         dst.write(chunk)                                                            # 写入每个分块
 _assert_video_file(tmp_abs, file.name or '')                                        # 深度校验：防止伪装文件（依赖 magic/filetype）
 os.replace(tmp_abs, video_abs)                                                      # 原子替换：确保最终文件完整
 
 width, height, duration = _probe_video(video_abs)                                   # ffprobe 探测：获取宽/高/时长
 thumb_exists = _make_thumbnail(                                                     # ffmpeg 截帧：生成封面
     video_abs,
     thumb_abs,
     ts_sec=max(1, int((duration or 0) // 2) if duration else 1),                     # 优先截取中间帧，否则取第 1 秒
 )
 
 v = Video.objects.create(                                                           # 创建视频记录：写入数据库
     title=(request.data.get('title') or os.path.splitext(file.name or '')[0] or '未命名视频')[:200],  # 标题：用户输入优先
     description=request.data.get('description') or '',                               # 描述：可为空
     video_file=video_rel[:100],                                                      # 视频路径（CharField），兼容旧逻辑
     thumbnail=thumb_rel[:100] if thumb_exists else None,                             # 封面路径（生成成功才写）
     video_file_f=video_rel[:200],                                                    # FileField 路径：用于 default_storage
     thumbnail_f=(thumb_rel[:200] if thumb_exists else None),                         # ImageField 路径：用于 default_storage
     duration=int(duration or 0),                                                     # 元数据：时长
     width=int(width or 0),                                                           # 元数据：宽
     height=int(height or 0),                                                         # 元数据：高
     file_size=int(file.size or 0),                                                   # 文件大小：用于统计/配额
     status='processing',                                                             # 状态：处理中（等待转码）
     upload_status='completed',                                                       # 上传状态：已完成
     user=request.user,                                                               # 归属用户：用于权限与作者信息
 ```

 在论文层面，为满足“协同过滤推荐算法实现”的描述，可在该基线推荐之上扩展 ItemCF（使用 Favorite/Like 行为构造物品共现），其核心公式为：
 
 \[
 sim(i,j)=\frac{|U_i \cap U_j|}{\sqrt{|U_i|\cdot|U_j|}}
 \]
 
 在线召回评分：
 
 \[
 score(u, j) = \sum_{i \in I_u} sim(i,j)
 \]
 
 其中 `U_i` 为与物品 i 发生过收藏/点赞行为的用户集合，`I_u` 为用户 u 的行为物品集合。该算法可通过离线任务定期计算 TopK 相似度并缓存到 Redis，以降低在线计算开销。
 
 **（工程化扩展）ItemCF：离线相似度计算 + 在线召回**
 
 - **数据来源**：`Favorite(user_id, video_id)`、`Like(user_id, video_id)` 两类行为可视为隐式反馈。
 - **目标**：对每个物品 `i` 计算最相似的 TopK 物品列表 `S_i`，并在请求推荐流时，基于用户历史 `I_u` 汇总召回候选。
 
 **清单 9：ItemCF 离线计算伪代码（可实现为 Celery beat 周期任务）**
 ```python
  # 输入：行为表 Favorite/Like
  # 输出：每个视频 i 的 TopK 相似视频列表，写入 Redis：itemcf:sim:{i} -> [(j, sim), ...]

 def build_user_item_sets():                                                         # 目标：构建用户->物品集合的倒排结构
     # user_id -> set(video_id)                                                      # 数据结构说明：每个用户对应其发生过正反馈的物品集合
     U = defaultdict(set)                                                            # 默认值为 set，便于去重与 O(1) 查询
     for (u, v) in Favorite.select(user_id, video_id):                               # 遍历收藏行为记录（伪代码：可用 ORM values_list 实现）
         U[u].add(v)                                                                 # 将收藏视频加入用户集合
     for (u, v) in Like.select(user_id, video_id):                                   # 遍历点赞行为记录
         U[u].add(v)                                                                 # 将点赞视频加入用户集合（与收藏并为同一类隐式反馈）
     return U                                                                        # 返回用户行为集合，用于后续共现统计

 def compute_item_cooccurrence(U):                                                   # 目标：统计物品共现次数与物品被多少用户触达
     # co[i][j] += 1  (i!=j)                                                         # 共现含义：同一用户同时喜欢 i 与 j 视为一次共现
     co = defaultdict(lambda: defaultdict(int))                                      # 二维稀疏矩阵：co[i][j] 存储共现计数
     cnt = defaultdict(int)  # cnt[i] = |U_i|                                        # cnt[i]：有多少用户与物品 i 产生过行为
     for u, items in U.items():                                                      # 遍历每个用户
         items = list(items)                                                         # 将 set 转 list，便于双重循环
         for i in items:                                                             # 统计每个物品的用户数
             cnt[i] += 1                                                             # cnt[i] += 1 表示多一个用户喜欢该物品
         for i in items:                                                             # 统计两两共现：同一用户的物品两两组合
             for j in items:
                 if i == j:                                                          # 排除自身共现
                     continue
                 co[i][j] += 1                                                       # 共现计数 +1
     return co, cnt                                                                  # 返回共现矩阵与物品用户计数

 def normalize_to_similarity(co, cnt, topk=200):                                     # 目标：将共现计数归一化为余弦相似度并取 TopK
     sim = {}                                                                        # sim[i] = [(j, s), ...]
     for i, row in co.items():                                                       # 遍历每个物品 i 的共现邻居
         arr = []                                                                    # 临时列表：存储 i 的候选相似物品
         for j, c in row.items():                                                    # 遍历每个邻居 j 及其共现次数 c
             s = c / math.sqrt(cnt[i] * cnt[j])                                      # 余弦相似度：c / sqrt(|U_i|*|U_j|)
             arr.append((j, s))                                                      # 记录 (物品 j, 相似度 s)
         arr.sort(key=lambda x: x[1], reverse=True)                                  # 按相似度降序排序
         sim[i] = arr[:topk]                                                         # 截断只保留 TopK，控制存储与在线读取成本
     return sim                                                                      # 返回物品相似度表

 def dump_to_redis(sim):                                                             # 写入缓存：将离线结果提供给在线推荐快速读取
     for i, arr in sim.items():                                                      # 遍历每个物品 i 的相似列表
         redis.set(f"itemcf:sim:{i}", json.dumps(arr), ex=24*3600)                   # 存 Redis：key=itemcf:sim:i，value 为 JSON，24h 过期
 ```
 
 **清单 10：ItemCF 在线召回伪代码（在推荐接口内进行候选生成）**
 ```python
 # 输入：用户 u 的历史行为集合 I_u（收藏/点赞），以及 Redis 中的 itemcf:sim:{i}
 # 输出：候选视频集合 candidates（再与新内容/热门内容混排）

 def recall_itemcf_candidates(user_id, K_items=50, K_sim=50):                         # 在线召回入口：给定 user_id 生成候选视频列表
     I_u = get_user_history_items(user_id)   # set(video_id)                          # 取用户历史正反馈集合（收藏/点赞），并去重
     scores = defaultdict(float)                                                     # 候选打分表：scores[j] 为候选 j 的累计相似度得分
     for i in list(I_u)[:K_items]:                                                   # 取前 K_items 个历史物品（控制在线复杂度）
         arr = redis.get(f"itemcf:sim:{i}")                                          # 读取物品 i 的相似 TopK（离线计算写入 Redis）
         if not arr:                                                                  # 缓存缺失：可能未计算或已过期
             continue                                                                 # 跳过该历史物品
         for (j, s) in json.loads(arr)[:K_sim]:                                       # 遍历前 K_sim 个相似物品
             if j in I_u:                                                             # 过滤用户已喜欢/已看过物品，避免重复推荐
                 continue
             scores[j] += float(s)                                                    # 累加得分：多历史物品共同指向会提升候选排名
     # 取分数最高的 TopN 作为候选
     return [j for (j, _) in sorted(scores.items(), key=lambda x: x[1], reverse=True)[:200]]  # 按得分降序取 Top200 候选
 ```
 
 **数据结构与复杂度分析**
 - **用户-物品集合**：`U[u] = set(video_id)`，构建复杂度约为 `O(|R|)`，`|R|` 为行为记录数。
 - **共现统计**：对每个用户的物品集合大小为 `d_u`，共现计算为 `O(Σ_u d_u^2)`；当 `d_u` 较大时需要：
   - 截断最近行为（例如只保留最近 100 个）
   - 或按时间窗口统计
 - **相似度归一化**：遍历共现边 `|E|`，复杂度 `O(|E|)`。
 - **在线召回**：若用户取 `K_items` 个历史物品，每个读取 `K_sim` 个相似物品，则复杂度约为 `O(K_items * K_sim)`，可控且适合在线。

 ```python
 class Video(models.Model):                                                          # 视频主表：承载媒体信息、元数据与状态

     class Meta:                                                                     # 数据库级优化与一致性约束

         ]
 ```
